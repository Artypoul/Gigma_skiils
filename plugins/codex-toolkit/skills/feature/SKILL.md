# /feature - фича от идеи до готового PR

Это skill не только для планирования: он ведёт полную разработку фичи в doc-first порядке. Сначала сделай документ-план, открой planning-only PR и дождись review-gate, затем напиши код в той же ветке/PR, проверь реализацию и доведи pull request до review-ready состояния. Пользователь мерджит в конце.

## Зачем именно так

Ошибка в дизайне дёшево чинится, пока это текст. Ошибка в готовом коде стоит дороже. Поэтому сначала зафиксируй план, проверь его, и только потом пиши код.

## Как общаться

Пользователь часто не разработчик. Пиши простым языком, короткими предложениями, без лишнего жаргона; термин при первом употреблении объясни одной строкой. В чат выноси только суть: что делаешь, что уже готово, что дальше. Технические детали держи в плане, PR и self-review.

**Язык артефактов фичи:** описание фичи, plan-doc в `docs/`, PR `Summary` и
`Test plan` пиши на русском. Английский оставляй только там, где он является
частью технического контракта: имена классов, API-полей, команд, ключей,
сообщений commit и принятых в репозитории терминов. Отступать от этого правила
можно только по явной просьбе пользователя или по более строгому локальному
правилу репозитория.

Перед стартом фичи убедись, что понятны желаемый результат и причина. Если глобальный clarification-вопрос уже задан и ответ пользователя это объясняет, используй этот ответ и не задавай второй вопрос. Иначе спроси: «Что ты хочешь сделать и почему?». После ответа кратко переформулируй цель, назови риски/развилки и предложи следующий шаг.

Если запрос, план или деплой выглядят спорно или опасно, с пользователем можно спорить, но только предметно: через факты из кода, DNS, конфигов, HTTP-ответов, логов, прав доступа, данных или продакшен-риска. Не спорь вкусом и не дави авторитетом; если уверенность ниже высокой или есть несколько нормальных продуктовых вариантов, задай конкретный уточняющий вопрос и предложи рекомендуемый вариант.

## Кто что делает

- Ты: собираешь контекст, пишешь план, пишешь код, прогоняешь проверки, разбираешь замечания.
- Плановый консилиум: 2-3 независимые review-роли помогают доработать plan-doc до кода. Роли выбирай по рискам задачи: product contract, architecture/data/migrations, API/RBAC, security, tests/maintainability, frontend UX.
- Монстры / PR-боты: Claude monster, Codex monster, `monster-reviewer`, `Codex monster review` или аналогичные ревьюеры проекта. Если они подключены, после каждого push их ревью считается частью workflow.
- Пользователь: ставит задачу, отвечает только на реальные продуктовые развилки и мерджит в конце.

## Совместимость с first-pass quality и project skills

`$first-pass-quality-gate` — защитная оболочка, а `$feature` и локальные project skills — владельцы рабочего процесса. Они не заменяют друг друга.

Приоритет правил: прямое указание пользователя и safety-ограничения → ближайший `AGENTS.md`/локальный project skill → `$feature` → общие значения first-pass quality. Более строгий локальный gate сохраняется; конфликт нельзя «решать» ослаблением безопасности.

Для feature используй отдельный Task Lock на каждую фазу:

1. `Workflow feature`, `WorkflowStage planning`: общий `Scope` — корень репозитория, `WriteScope` — только plan-doc и явно разрешённые planning-файлы; `PrePublishWhen` описывает проверенный план перед commit/push/PR, `DoneWhen` — опубликованный planning-only PR без незакрытых плановых блокеров.
2. После снятия plan review-gate заверши planning Task Lock и создай новый через `-Continuation`: `WorkflowStage implementation`, `WriteScope` — только утверждённый кодовый scope; `PrePublishWhen` — тесты и self-review до push, `DoneWhen` — актуальный PR без блокеров по текущему diff.
3. Перед `commit`, `push` и созданием/обновлением PR передай `publish` и `selfReview` gates. Обычные `git add/commit/push` и PR-операции не обнуляют проверку уже проверенного содержимого; новая правка файла обнуляет `publish`, acceptance, self-review и review.
4. `git push`, создание/правка/комментарий/review PR — dev-workflow в `pr` mode. Merge, force-push, deploy и production-write остаются production-действиями и требуют отдельного точного разрешения.

Не используй sub-agents автоматически только ради «консилиума». Если пользователь не просил делегирование, используй monster-review, локальные review skills или независимый ручной self-review ролями. Если sub-agents явно разрешены, они всё равно требуют bounded handoff и независимой проверки родительским агентом.

## Поток

### Шаг 0 - Подготовка

1. Если пользователь ещё не объяснил цель и причину в уже заданном clarification-ответе, спроси: «Что ты хочешь сделать и почему?». Не дублируй вопрос.
2. Прочитай локальные правила репозитория: `AGENTS.md`, `CLAUDE.md`, `.claude/skills/`, `.codex/skills/`, `README` и соседние документы. Канон репозитория важнее общих привычек.
3. Быстро собери контекст по затронутым модулям, тестам, данным, API и пользовательскому сценарию.
4. Определи базовую ветку по правилам проекта. Если явного правила нет, опирайся на текущий git flow.
5. Создай ветку `feature/<short-name>`, если проект не требует другой схемы именования.
6. До первой planning-правки создай planning Task Lock по правилам совместимости выше и передай workflow gate после загрузки локальных project skills.
7. Если задача на самом деле маленькая и безопасная, не используй этот skill: просто сделай правку.

Всегда полный флоу с планом: план → ревью → код → ревью. Что достойно отдельной фичи, определяет пользователь и локальные правила репозитория; совсем мелкие безопасные правки делаются вне этого skill.

### Шаг 1 - План-документ

1. Напиши короткий план в `docs/plan-<feature>.md` или в той папке, которую предпочитает репозиторий.
2. Зафиксируй в плане:
   - что делаем и зачем;
   - что входит и не входит в scope;
   - какие файлы, модули, эндпоинты, таблицы или экраны трогаем;
   - **что будет в БД — обязательно схемой:** если БД меняется или фича хранит/читает новые данные, покажи таблицы, новые/изменённые поля, индексы, связи, где лежат токены/секреты, пример строки данных и что намеренно НЕ создаётся. Если БД не трогаем, прямо напиши `БД не меняется`. Формат — markdown-таблица, Mermaid ERD или короткий блок `таблица -> поля -> связи`;
   - **что переиспользуем (проверено по коду) — НЕ изобретай заново:** до проектирования новой таблицы/сервиса/эндпоинта прогрепай репозиторий на дубли. Логика — есть ли сервис/команда/джоба/контроллер, который уже делает похожее. Таблицы и поля — есть ли таблица/колонки, где данные уже живут (миграции, `$fillable`, поля «внешних ID» типа `provider_id`/`provider_code`, `*_integrations` + параметры). Нашёл — реюз/расширяем, не пиши параллель; по каждому решение «реюз / расширяем / новое — почему»;
   - edge cases, риски и совместимость;
   - как будет проверяться результат.
3. Сразу после первого draft plan-doc собери плановый консилиум на доработку документа. Минимум 2-3 роли по рискам задачи; используй monster-review, локальные project skills или ручной self-review ролями. Sub-agents допустимы только когда пользователь или более приоритетная активная инструкция явно разрешили делегирование.
4. Внеси реальные замечания консилиума в plan-doc до открытия/обновления planning PR. В плане или PR body явно укажи: какие роли смотрели план, какие блокеры/риски нашли, что исправлено, что осталось продуктовым вопросом.
5. Если в репозитории уже есть шаблоны планов или локальные skills вроде `backend-plan` или `frontend-plan`, используй их вместо собственного формата.
6. Как только план-документ и первый плановый консилиум готовы, создай commit, сделай push и открой PR сразу как `ready for review`, не `draft`. На старте в PR должен быть только план-документ или минимальный набор файлов, нужный для обсуждения плана.
7. В `PR body` явно напиши, что это planning-only раунд: в этом PR сейчас только план, реализации пока нет, код пойдёт после апрува плана. Не полагайся только на комментарий в таймлайне: ревьюеры и боты должны увидеть это прямо в описании PR.
8. Оформляй этот первый `PR body` в явном формате с разделами `Summary` и `Test plan`.
9. В `Summary` кратко перечисли scope плана, ключевые решения текущего раунда и результат планового консилиума.
10. В `Test plan` укажи `not run (planning document only)`, если в PR пока только документация и проверять ещё нечего.
11. Не затирай уже существующие записи в PR без явной причины. Если нужно обновить `PR body`, правь его узко и сохраняй уже добавленные человеком или агентом секции, заметки и контекст. Предпочитай дописывать новый статус или новый блок, а не переписывать всё описание с нуля.
12. Не удаляй историю обсуждения: не компенсируй новый контекст массовой перезаписью комментариев, не скрывай прошлые договорённости и не стирай старые summary, если они всё ещё полезны для ревью.
13. Делай коммиты последовательно и прозрачно: каждый заметный шаг, правка по ревью или переход план -> код должен идти новым commit'ом с понятным сообщением.
14. Не используй `git commit --amend`, rebase с переписыванием опубликованной истории или force-push в feature workflow, если пользователь этого прямо не попросил.
15. Если задача упирается в реальное продуктовое решение пользователя, остановись после плана и спроси только про этот узкий выбор. Иначе переходи дальше без паузы.
16. Сразу после открытия planning-only PR считай, что включён review-gate: до его снятия нельзя писать production code. Разрешены только правки plan-doc, PR body и ответы на замечания к плану.
17. После создания PR сразу пришли пользователю полную ссылку на PR. После каждого последующего push/update этого PR снова укажи ссылку в ближайшем статусе или финальном сообщении.

После успешного push watch-hook только ставит pending-маркер; он не читает GitHub и не является поллером. Сам агент или реальный monitor должен проверять ревью через GitHub/CLI и inbox до снятия review-gate.

### Шаг 2 - Проверка плана

1. Проверь план на дыры до написания кода: права доступа, пустые состояния, миграции, ошибки, откат, наблюдаемость, тесты.
2. Проведи или обнови плановый консилиум по опубликованному plan-doc. Если план менялся после замечаний, консилиум должен смотреть свежую версию, а не старый draft.
3. Если в проекте есть автоматическое ревью PR, PR-боты или monster-пайплайн, используй их. Для plan-only PR обязательно дождись Claude monster и Codex monster / `monster-reviewer` и `Codex monster review`, если они подключены в проекте. Их копии часто приходят в `$HOME/.claude-bot-inbox/<owner>-<repo>-pr<N>.md`.
4. Пока не получен сигнал review-gate, не пиши код фичи, не меняй тесты под реализацию и не начинай implementation commits. На этом этапе разрешены только правки плана, PR body и review-related updates.
5. Исправь блокирующие замечания в плане в том же PR отдельными понятными commit'ами, не затирая уже опубликованную историю без прямой команды пользователя.
6. После исправления блокеров плана обнови вывод планового консилиума: что было найдено, что исправлено, какие риски остались и почему они не блокируют код.
7. Считай review-gate снятым только если выполнено одно из условий:
   - пользователь явно написал, что план одобрен и можно идти в код;
   - плановый консилиум и `monster-reviewer` / `Codex monster review` отработали, и по их замечаниям не осталось блокеров;
   - пользователь явно разрешил не ждать внешнее ревью.
8. Если есть неоднозначность между «не затягивай» и «ждать ревью», всегда выбирай ожидание ревью.
9. Если после 4 кругов ревью по плану блокеры не заканчиваются, остановись и коротко позови пользователя.

Чини все реальные замечания P1/P2/P3, кроме вкусовщины и вопросов, где нужно решение владельца продукта. Вердикт «блокеров нет», «можно мержить», «Swish» или «no major issues» завершает цикл; не запускай бесконечные дополнительные круги «на всякий случай».

### Шаг 2.5 - Wait For Review Gate

1. После создания planning-only PR остановись и дождись review-cycle по плану.
2. Сначала проверь плановый консилиум, затем `monster-reviewer`, затем `Codex monster review`, затем остальные PR comments/reviews, если они есть.
3. Пока review-gate не снят, делай polling статуса ревью каждые 30 секунд. Проверяй `monster-reviewer`, `Codex monster review`, review comments, reviews и обычные comments в PR.
4. Если ревью пришло, разбирай только замечания к плану и обновляй только документ, PR body и planning commits.
5. Если ревью ещё не пришло, не начинай код без явного разрешения пользователя или явного правила проекта, что ждать не нужно.
6. При любом сомнении выбери более строгий режим: ждать review-gate, а не переходить к реализации.

### Шаг 3 - Переход план -> код

Как только по плану нет блокеров И review-gate снят, зафиксируй `review=passed`, заверши planning Task Lock и создай implementation Task Lock через `-Continuation`. Затем начинай реализацию сам, без лишней паузы и без нового запроса. План остаётся в PR как рабочее ТЗ, по которому потом сверяют реализацию.

### Шаг 4 - Реализация

1. Реализуй фичу строго по утверждённому плану в той же ветке.
2. Следуй существующим паттернам проекта и переиспользуй локальные helper'ы, тестовые утилиты и инфраструктуру.
3. Держи изменения сфокусированными на задаче.
4. Доводи фичу до production-ready состояния: закрой основные состояния, ошибки, пустые случаи и совместимость.
5. Commit -> push.
6. После каждого push с кодом обязательно отпишись в самом PR комментарием: что изменено, какие проверки прошли, что ещё красное/ожидает, и нужен ли новый review. Не ограничивайся только сообщением в чат.
7. После push напомни пользователю полную ссылку на текущий PR, чтобы он не искал её в истории чата.

### Шаг 5 - Проверка кода

1. Запусти релевантные тесты, линтеры, форматтеры и статические проверки.
2. Просмотри diff как ревьюер: ищи баги, регрессии, пропущенные состояния, лишний scope и неясные названия.
3. После реализации и локальных проверок собери консилиум по фиче: минимум 2-3 независимых review-роли по рискам задачи (например API/RBAC, product contract, tests/maintainability, frontend UX, security). По умолчанию используй monster-review, локальные review skills или ручной self-review; sub-agents — только при разрешённом делегировании. Вывод должен быть разобран до финала.
4. Если monster-review, консилиум или другой reviewer нашёл блокеры или реальные P1/P2/P3 — исправь их в той же ветке/PR, снова прогони релевантные проверки, сделай push и повторно пришли пользователю ссылку на PR.
5. После каждого исправления по review обязательно заново запусти консилиум/re-review по текущему diff. Нельзя считать PR готовым, если последний консилиум смотрел старый diff, plan-only состояние или код до починки.
6. Если внешний monster-review, Codex review или Claude review завис, недоступен или упёрся в лимиты, не подменяй это молчаливым "готово": явно скажи о недоступности и запусти локальный консилиум ролями по актуальному diff.
7. Если в проекте есть внешний цикл ревью, дождись замечаний и разбери их до состояния без блокеров.
8. Повторяй цикл fix -> push -> re-review -> consilium, пока блокеров не останется, и сохраняй историю последовательными commit'ами вместо переписывания уже опубликованной ветки.
9. Если прошло 4 круга, а блокеры продолжают возвращаться, остановись и коротко опиши пользователю, что осталось.

«Бот сказал OK» — это финал review-цикла, а не предварительный кивок. Не догоняй бесконечные мелочи после снятия блокеров, если они не меняют production-риск.

### Шаг 6 - Финал

1. Если в репозитории есть `/pr-finalize` или аналогичный локальный workflow, прогони его: человеческое описание PR, сценарии «до/после», тест-план, HISTORY/журнал, self-review.
2. Подготовь понятное summary для PR или пользователя: что изменилось, что теперь работает иначе, как это проверено.
3. Если workflow проекта предполагает PR, собери review-ready описание без кода и с понятными сценариями проверки.
4. Сообщи пользователю, что задача готова к мерджу, и дай полную ссылку на PR, если она есть.
5. Не мержи без прямой команды пользователя.

### Шаг 7 - После мерджа

Если пользователь пишет `смержил`, `смерджил`, `merged` или явно говорит, что PR уже влит:

1. Проверь, что PR действительно merged в целевую ветку и что локальный/remote `master` видит нужный merge commit.
2. Сделай production sanity-check: нет ли явных красных блокеров, которые прямо делают деплой опасным. Если риск критический и подтверждённый, остановись и коротко объясни.
3. Слова `смержил`/`merged` подтверждают только факт merge и никогда сами по себе не разрешают deploy. Подготовь точный deploy-путь, целевой environment и commit, затем попроси отдельное явное подтверждение, если последняя реплика пользователя уже не содержит недвусмысленного разрешения на этот конкретный deploy.
4. После отдельного подтверждения создай high-risk production Task Lock, Entity Lock на точный deploy wrapper/input и выполни один разрешённый deploy-вызов. Не используй raw SSH/shell, если нет специально разрешённого typed project wrapper.
5. После успешного deploy запусти end-to-end проверку: проверь живой API/основной пользовательский сценарий фичи на указанном production или staging-пути.
6. В финале коротко напиши: какой commit задеплоен, как прошёл deploy, какие e2e-проверки выполнены и что осталось красным/непроверенным. Не называй deploy успешным при unknown outcome.

Пользователь главнее workflow. Если пользователь пишет «стоп», «оставь», «не туда», `pause` или `stop`, остановись немедленно, не пытайся «дочинить круг». Хуки/турникет держат агента, не пользователя.

## Рельсы feature workflow

При активном plugin `first-pass-quality` его controller является каноническим enforcement-слоем: `Workflow feature` требует stage, успешный push заново открывает `review`, а `ready` блокируется до свежего review. Он не заменяет смысл workflow, но не даёт тихо перескочить через ревью-цикл.

Отдельные глобальные feature-hooks ниже — только legacy/fallback для среды без `first-pass-quality`. Не регистрируй оба механизма для одной сессии: это дублирует hook-вызовы и сообщения.

- `$HOME/.codex/hooks.json`
  - `PostToolUse`: watch-хук ставит pending-маркер только после доказанно успешного `git push`/`gh pr create` в активном `Workflow feature`; без first-pass state используется осторожный fallback только для веток `feature/*`/`feat/*`.
  - `Stop`: turnstile-хук не даёт закончить ход, пока pending-маркер существует.
  - `UserPromptSubmit`: если пользователь явно пишет `стоп`, `оставь`, `не туда`, `pause` или `stop`, ставится stop-флаг для обхода turnstile.
- Маркеры лежат в `$HOME/.codex/feature-hooks/`.
- Глобальный выключатель обоих хуков: `$HOME/.codex/feature-hooks-off`.

Claude-версия этого workflow может использовать хуки в `$HOME/.claude/settings.json` (`feature-watch.ps1`, `feature-turnstile.ps1`). В Codex legacy/fallback-аналогами являются `codex-feature-watch.ps1`, `codex-feature-user-prompt.ps1`, `codex-feature-turnstile.ps1`.

Если turnstile остановил завершение хода:

1. Проверь свежие ревью и разбери замечания.
2. Если в проекте используется твой текущий monster-пайплайн, смотри inbox в `$HOME/.claude-bot-inbox/<owner>-<repo>-pr<N>.md`.
3. Когда review-цикл реально завершён и PR готов, выставь `review=passed` в first-pass quality. Turnstile сверит state и удалит маркер автоматически; без активного first-pass state допускается удалить только точный session marker, названный хуком.
4. Если пользователь явно остановил задачу, stop-флаг обычно ставится автоматически через `UserPromptSubmit`.

## Правила выбора

- Предпочитай этот skill для нетривиальных, многосоставных или рискованных задач.
- Предпочитай короткий план перед кодом, если меняется API, UI, схема данных, фоновые процессы или несколько подсистем сразу.
- Если локальные инструкции проекта строже этого global skill, следуй локальным инструкциям.
- Если пользователь просит пропустить план, соглашайся только для простых задач. Для сложных задач всё равно сделай короткий план, но не растягивай его.
