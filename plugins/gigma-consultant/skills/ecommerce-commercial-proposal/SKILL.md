# Коммерческое предложение на интернет-магазин (КП)

Цель — собрать из брифа (`ecommerce-discovery`) и решения (`ecommerce-solution-fit`) предложение, которое клиент **поймёт без программиста** и сможет согласовать.

Опора по возможностям — [`reference/ecommerce-capabilities.md`](../../reference/ecommerce-capabilities.md).

## Принципы КП

- **Человеческим языком.** Сценарии «сейчас у вас так → станет так», а не список endpoint'ов. Технические детали — отдельным мелким блоком «под капотом» в конце, если вообще нужны.
- **Честный объём.** Включать только то, что Gigma реально умеет (карта возможностей). Помеченное ⚠ — либо обходом с пояснением, либо не обещать.
- **Стоимость не выдумывать.** Реальной тарификации Gigma в API/доках нет. Цены/тариф берутся из прайса Gigma или согласуются с владельцем — в КП это поле **«согласовать»**, а не придуманное число. Можно структурировать (разовая настройка + витрина + сопровождение), но суммы — заглушка до согласования.
- **Разделять зоны.** Что делает Gigma (бэкенд/настройка), что — отдельный проект (витрина-сайт, дизайн), что — клиент (домен, кабинет YooKassa, прайс, фото).

## Структура КП (шаблон)

```markdown
# Коммерческое предложение: интернет-магазин «<клиент>» на Gigma

## 1. Задача
Коротко, словами клиента: что продаёт, что болит сейчас.
(из брифа: главная боль, «успех через 1–2 месяца»)

## 2. Что предлагаем
1–2 абзаца: онлайн-магазин на платформе Gigma — каталог, корзина, оплата, склад,
кабинет клиента, бэк-офис для ваших сотрудников. «Движок» Gigma + витрина-сайт на вашем домене.

## 3. Что вы получите (сценарии «до → после»)
- Каталог: сейчас ___ → товары в системе, цены обновляются загрузкой прайса.
- Заказы: сейчас ___ → клиент сам оформляет на сайте, заказ падает в бэк-офис.
- Оплата: сейчас ___ → оплата картой онлайн (YooKassa) / при получении.
- Склад: сейчас ___ → остатки в системе, видно что есть в наличии.
- (взять реальные пункты из solution-fit)

## 4. Состав работ (этапы)
1. Завести проект (тенант) и структуру (бизнес, реквизиты, роли).
2. Каталог: категории + загрузка прайса + фото.
3. Склад и остатки.
4. Оплата (подключение YooKassa) и доставка.
5. Контент витрины (страницы, баннеры, меню).
6. Витрина-сайт + домен, тестовый заказ, запуск.
(детализация — в плане внедрения, скил ecommerce-launch-plan)

## 5. Что входит / не входит
Входит: настройка Gigma, загрузка каталога/остатков, оплата, доставка, контент, тестовый прогон.
Не входит (обсуждается отдельно): дизайн и разработка самого сайта-витрины, домен,
кабинет YooKassa (заводит клиент), наполнение текстами/фото сверх предоставленного.

## 6. Что нужно от вас
- Прайс файлом (Excel или выгрузка 1С).
- Фото товаров.
- Реквизиты компании и доступ/кабинет YooKassa для оплаты.
- Домен (или поможем подобрать).
- Контакт для согласований.

## 7. Сроки
Ориентир по этапам (из плана внедрения). Уточнить под объём каталога.
⚠ Большой каталог грузится пачками — заложить время.

## 8. Стоимость
| Позиция | Сумма |
|---|---|
| Разовая настройка и запуск | согласовать |
| Платформа Gigma (тариф) | согласовать по прайсу Gigma |
| Сопровождение (опционально) | согласовать |
> Точные суммы — после согласования объёма и тарифа Gigma.

## Под капотом (по запросу, техническому контакту)
Headless-архитектура: API api.gigma.ru + витрина-сайт + YooKassa.
Контракты — erp-rules.txt / openapi.json.
```

## Перед отправкой — самопроверка

- [ ] Нет обещаний из ⚠/❌ карты возможностей без оговорки.
- [ ] Стоимость не выдумана — стоит «согласовать», где нет утверждённого прайса.
- [ ] Чётко разделено: Gigma / витрина-сайт / клиент.
- [ ] Сроки учитывают объём каталога и лимит загрузки.
- [ ] Язык — для не-программиста; техника убрана в «под капотом».

## Что дальше

КП согласовано → скил **ecommerce-launch-plan**: разложить состав работ в пошаговый запуск и передать в операционные скилы.
