# Подбор решения на Gigma под интернет-магазин

Цель — превратить бриф (из `ecommerce-discovery`) в понятную клиенту картину: что именно он получит, как это работает и где границы. Без КП и цен — это шаг про **ценность и устройство**.

Опора — [`reference/ecommerce-capabilities.md`](../../reference/ecommerce-capabilities.md). Если нужен точный контракт — сверься с `erp-rules.txt`/`openapi.json` (ссылки в карте). **Не добавляй возможностей, которых нет в карте.**

## Шаг 1. Построить матрицу «надо → есть в Gigma»

По каждому пункту брифа — строка. Честно проставляй статус.

| Потребность клиента | В Gigma | Как работает (просто) | Ограничение, о котором сказать |
|---|---|---|---|
| <из брифа> | ✅ есть / ⚠ частично / ❌ нет | <одно предложение по-человечески> | <если есть> |

Опорные соответствия (взять нужные строки):

- **Каталог** → номенклатура (товары/услуги) + категории + бренды; прайс грузится файлом Excel, повторная заливка обновляет цены.
- **Остатки/склад** → склады + остатки; можно завести несколько складов; загрузка остатков файлом или из 1С (JSON-накладная).
- **Корзина и оформление** → витрина считает сумму (precalculate) и оформляет заказ. ⚠ телефон берётся из профиля клиента, отдельно не передаётся.
- **Оплата картой** → YooKassa: клиента уводит на платёжную страницу, после оплаты заказ сам становится «оплачен». Нужен **кабинет YooKassa клиента**. Свои реквизиты можно на каждый склад (удобно для мультимагазина).
- **Оплата наличными/при получении** → отдельный способ оплаты, без YooKassa.
- **Доставка** → самовывоз (привязка к магазину-точке) и доставка по адресу (адрес подсказывается через Dadata).
- **Личный кабинет** → вход по телефону, профиль, избранное, история заказов.
- **Промокоды** → скидки с промокодами (есть статистика, ссылки-приглашения). ⚠ «промоакции» как отдельная фича сейчас нестабильны — используем скидки.
- **Контент витрины** → страницы (о доставке/оплате), баннеры-слайдер, меню, динамические блоки — наполняются из бэк-офиса.
- **Роли сотрудников** → роли и права; каждый видит только свой проект.

## Шаг 2. Честно отметить ограничения (проговорить клиенту)

Не прятать — называть как «особенности, которые учтём»:

- **Уведомления о статусе** — витрина их подтягивает опросом (раз в ~минуту), мгновенных пушей по событию наружу нет.
- **Массовые операции** — заливка большого каталога идёт пачками (примерно 1 позиция в секунду из-за лимита), большой прайс = ощутимое время; это нормально, закладываем в план.
- **Заказы со стороны бэк-офиса** оформляются через витринный контур (так надёжнее) — на функциональность клиента не влияет.

Если в брифе стоит флаг «запрошено что-то ⚠/❌» — здесь явно объясни обходной путь или скажи, что пока нет.

## Шаг 3. Нарисовать архитектуру (текстом, просто)

Gigma — **headless**: «движок» отдельно, «витрина» отдельно.

```
[ Покупатель ]
      │  открывает сайт-витрину (ваш домен)
      ▼
[ Витрина-сайт ]  — отдельный фронт (статика/Next), хостинг + домен
      │  спрашивает товары, считает корзину, оформляет заказ
      ▼
[ Gigma API ]  api.gigma.ru  — каталог, склад, заказы, клиенты, оплата
      │
      ├── оплата картой → YooKassa (кабинет клиента) → деньги клиенту
      └── бэк-офис Gigma — где сотрудники ведут товары, склад, заказы
```

Что отсюда нужно проговорить клиенту:
- **Витрина-сайт** — отдельный проект (дизайн + фронт). Gigma даёт «движок» и данные, но не рисует сам сайт. Уточни: есть ли у клиента сайт/дизайн или нужен с нуля.
- **Домен** — нужен домен клиента (или подбираем).
- **Оплата** — серверная часть для платежей; кабинет YooKassa заводит клиент.

## Шаг 4. Сохранить решение

Запиши результат в `solution-<клиент>.md`: матрица + ограничения + схема. Это вход для `ecommerce-commercial-proposal`.
