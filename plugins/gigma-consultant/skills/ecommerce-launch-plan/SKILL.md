# План внедрения интернет-магазина на Gigma

Цель — развернуть всё для приёма онлайн-заказов и передать конкретные действия операционным скилам. Это **мост** от консультации к проду.

Опора: [`reference/ecommerce-capabilities.md`](../../reference/ecommerce-capabilities.md) + канонический `erp-rules.txt` (сценарий §17.3 «Заполнить витрину», тела §18, баги §19). Точные тела запросов — **всегда из `openapi.json`/`erp-rules.txt`, не из памяти.**

> Перед каждым реальным write-действием: показать клиенту/владельцу, что создаём, спросить **одним блоком**. POST не идемпотентны — повтор плодит дубли. Ожидать `2xx` (201 на create, 204 на delete), не строго 200.

## Авторизация перед запуском

Шаг 1 (`POST /api/projects`) **публичный**, токен не нужен. **Все шаги 2–13 требуют токена сотрудника** проекта: `POST /api/send_password` → `POST /api/login` → токен `<id>|<secret>` (код живёт 5 минут; если письмо не дошло — fallback из таблицы `passwords`, см. скил load-nomenclature §3). Не потерять `|` в токене.

## Порядок запуска

| # | Шаг | Чем делаем | ⚠ Ловушки / проверки |
|---|---|---|---|
| 1 | **Создать тенант (проект)** | скил **create-tenant** (gigma-erp): `POST /api/projects`, тело `{name, login, menu_id:7}` | `login` (email владельца) глобально уникален; `menu_id:7` обязателен; тенант создаётся пустым; тело — из create-tenant |
| 2 | **Бизнес + реквизиты** | `POST /api/branches` → `POST /api/branches/{id}/bank_requisites` | branches required: `title,name,inn,phone_1,email,address,legal_address`; реквизиты автозаполняются через `POST /api/search_bank` (по БИК) |
| 3 | **Магазин(ы) / точки выдачи** | `POST /api/shops` | required: `name,address,phone`; можно задать часы работы и праздники; самовывоз привязывается к `shop_id` |
| 4 | **Склад(ы)** | `POST /api/warehouses` | required: `name,address,storage_unit_id` (из `GET /api/storage_units`, **НЕ** `/api/units`); ⚠ create-tenant предупреждает, что складу нужен `city_id` (из `GET /api/cities`) — тело сверить с `openapi.json`/§18.0 перед созданием |
| 5 | **Категории каталога** | `POST /api/categories` | required: `name` (min 3); вложенность через `parent_id` |
| 6 | **Каталог (товары/услуги)** | скил **load-nomenclature** (gigma-erp): `POST /api/nomenclatures/import` — `multipart/form-data`, поле `file` (xls/xlsx/csv) | рекомендуется xlsx (на CSV импорт путается на разделителе); пустые ячейки `None`, не `''`; единице нужен `abbreviation`; повторная загрузка обновляет по артикулу (детали — в load-nomenclature) |
| 7 | **Остатки на складе** | `POST /api/inventories` поштучно или `POST /api/inventories/upload` (файл / JSON-накладная) | inventory required: `nomenclature_id,warehouse_id`; `counterparty_id` — только для поставщика (тип 2) |
| 8 | **Фото товаров** | `POST /api/files` (`file_type_id=2`) → `avatar_id` в `PUT /api/nomenclatures/{id}` | multipart; PUT шлёт **весь объект** (сначала GET, потом merge), иначе затрёшь поля |
| 9 | **Оплата** | YooKassa-creds на склад (`warehouse_integrations`); способы оплаты | карта = `payment_type_id=2`, наличные = `1`; webhook `POST /api/yookassa` сам ставит статус `22` (оплачен); кабинет YooKassa — у клиента |
| 10 | **Доставка** | `delivery_types` + subtypes + params | самовывоз (type 1 → `shop_id`), доставка по адресу (type 2 → subtype 2 → `address`); адрес — Dadata `/api/counterparty/search_address` |
| 11 | **Контент витрины** | страницы: `POST /api/pages` (по `slug`); блоки: `POST /api/applications/{id}/blocks`; навигация: `POST /api/applications/{id}/menu_items` | ⚠ слайды/баннеры (`GET /api/counterparty/slides`) — read-only в API, заводятся через UI Itecho; программного create для слайдов нет |
| 12 | **Витрина-сайт + домен** | отдельный фронт + хостинг + домен + серверный рендер для оплаты | Gigma не рендерит сайт — это **отдельный проект вне этих скилов**; оплате нужен SSR; уточнить у клиента: есть сайт/дизайн или делаем с нуля |
| 13 | **Тестовый заказ (smoke)** | сценарий §17.1 на тестовом клиенте | `POST /api/counterparty/login` → `GET /api/counterparty/products` → `POST /api/counterparty/orders/precalculate` → `POST /api/counterparty/orders` → `payment_link`; дождаться статус `22` (оплачено) опросом каждые ≥5 сек |

## Ключевые ловушки этого процесса (из §18–§19)

- **Тела шагов 3, 4, 6, 7** (магазины/склады/номенклатура/остатки) в каноне помечены ⚠ unverified — перед write сверять тело с `openapi.json`/§18, не считать схему 100% готовой.
- **Заказы оформлять через витринный контур** `POST /api/counterparty/orders` — он создаёт резервации сам, внутри транзакции. ERP-вариант `POST /api/orders/{id}/nomenclatures` сейчас баговый (500).
- **Отмена заказа** — не `DELETE /api/orders/{id}` (баг 500), а `PUT /api/orders/{id}` со статусом `6`.
- **Скидки** — через `discounts`, не `/api/promotions` (баг 500).
- **Контекст заказа.** Витринный заказ привязывается к `application_id`/`sales_channel_id`, которые middleware проставляет по токену клиента — тестовый клиент должен логиниться в контуре нужного проекта, иначе заказ не пройдёт.
- **Массовая заливка** — лимит ~60 запросов/мин: каталог и остатки грузятся пачками (≤1 в секунду, ≤5 параллельно). Большой прайс = заметное время, показать прогресс, уметь продолжить после обрыва (см. ETL-сценарий §17.7).
- **Пустой стенд** — на свежем тенанте справочники могут быть пустыми; проверить `count` перед массовыми операциями.
- **Bearer-токен** — целиком `<id>|<secret>`, не потерять `|`.

## Передача в «руки» (handoff)

Исполнение write-операций (curl/Bash, доступ к боевой БД для проверки) делают **операционные скилы плагина `gigma-erp`**, не консультант. Плагин `gigma-erp` должен быть установлен рядом — без него «руки» недоступны.

1. Шаг 1 → скил **create-tenant** (`gigma-erp`).
2. Шаг 6 → скил **load-nomenclature** (`gigma-erp`).
3. Остальные шаги — по `erp-rules.txt` (§17.3, §18) и `openapi.json`; перед write сверять тело запроса со схемой.
4. Проверка результата в БД требует VPN-доступа (нода `31.77.160.204` → priscilla `161.104.46.126`, `gen_user`) — детали в скилах create-tenant / load-nomenclature.
