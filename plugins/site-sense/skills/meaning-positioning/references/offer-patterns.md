# Offer patterns

Use these patterns as starting points, not as copy to paste blindly.

## Clear category offer

For products where the category is already familiar.

```text
[Category] for [segment] who need [outcome] without [main friction].
```

Example structure:

- what it is;
- who it is for;
- key result;
- risk/friction removed.

## Problem-first offer

For users who feel pain but do not know the solution category.

```text
Stop losing [cost] to [problem]. [Product] helps [segment] [new way] so they can [outcome].
```

## Mechanism-first offer

For crowded categories where the method is the difference.

```text
[Outcome] through [unique mechanism], not [weak alternative].
```

## Proof-led offer

For skeptical markets or expensive B2B decisions.

```text
[Result/proof] for [similar customers] — now available as [product/service].
```

## Speed / simplicity offer

Use only when speed or simplicity is proven.

```text
Go from [messy before] to [usable after] in [time/process], without [usual complexity].
```

## Risk-reversal offer

For high-friction purchases.

```text
Try/implement [solution] with [risk reducer]: [guarantee, pilot, audit, migration support, SLA].
```

## Local/service offer

For local services or expert providers.

```text
[Service] for [audience] in [location/context] — with [specific proof/process] and [clear next step].
```

## Rules

- The offer must name either audience, problem, outcome, mechanism, or proof.
- Do not use «комплексные решения» unless the scope is defined.
- Do not call something «уникальное» unless the difference is visible.
- One offer = one main promise. Supporting details belong below.
