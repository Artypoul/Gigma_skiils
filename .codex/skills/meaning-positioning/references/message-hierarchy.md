# Message hierarchy

The hierarchy prevents a page from becoming a list of disconnected claims.

## Basic hierarchy

```text
Hero claim
  → Subclaim / explanation
  → Benefit pillars
  → Mechanism
  → Proof
  → Objections
  → CTA
```

## What each layer must do

| Layer | Job | Common failure |
|---|---|---|
| Hero claim | Identify product, audience, outcome | vague slogan |
| Subclaim | Explain how/why it works | feature dump |
| Benefit pillars | Show concrete outcomes | generic benefits |
| Mechanism | Make the value believable | missing causal logic |
| Proof | Support strong claims | proof appears too late |
| Objections | Reduce risk | FAQ repeats marketing |
| CTA | Move to next step | «Узнать больше» everywhere |

## Hero decision rule

Choose the hero angle by what the audience needs first:

- Confused audience → category clarity.
- Skeptical audience → proof-led.
- Pain-aware audience → problem-first.
- Competitive market → differentiation/mechanism.
- Low-friction product → speed/simplicity.

## Claim ladder

```text
Feature → capability → user outcome → business result → proof
```

Example:

```text
Feature: автоматические отчёты
Capability: команда видит статус без ручного сбора данных
Outcome: меньше встреч и уточнений
Business result: быстрее закрываются проекты
Proof: минус 30% ручных отчётов в кейсе X
```

## Sanity checks

- Can the user understand the page if they read only headings?
- Does each section support the hero, or does it introduce a new strategy?
- Is every strong claim either proven or softened?
- Does the CTA match the stage of intent?
