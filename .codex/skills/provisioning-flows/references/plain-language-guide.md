# Простая памятка о provisioning flows

Provisioning — создание или подготовка сущности в другой системе. Сущностью может быть пользователь, подписка, сервер, заказ, группа, тариф или карточка клиента.

## Базовые операции

- **Get-or-create**: найти по устойчивому ключу; если записи точно нет — создать.
- **Upsert**: найти и обновить, а если записи нет — создать.
- **Reconciliation**: сравнить «как должно быть» с «как есть» и исправить разницу.

Перед созданием нужно выбрать устойчивый ключ: неизменяемый внешний ID, номер заказа или другую уникальную комбинацию. Номер телефона и email могут меняться, поэтому это только дополнительные признаки для поиска, а не основной ключ. Ошибка сети не подтверждает отсутствие записи.

## Подходы

| Подход | Когда применять | Риск |
| --- | --- | --- |
| Ручное создание | Редкие записи и админка | Ручные ошибки |
| Предварительное создание | Сущность должна быть готова заранее | Неиспользуемые записи |
| JIT / по требованию | Объект нужен не каждому | Дубли при неверной обработке ошибок |
| После подтверждённого события | Оплата, договор, одобрение | Задержки и повторы события |
| Upsert при синхронизации | Каталоги, клиенты, остатки | Нужен надёжный внешний ключ |
| Reconciliation | Доступы, группы, настройки | Исправление не мгновенно |

## Примеры других сущностей

| Сущность | Как применяется flow | Устойчивый ключ |
| --- | --- | --- |
| Клиент | Найти клиента в CRM при новом заказе; создать только при подтверждённом отсутствии | `counterparty_id` |
| Заказ | При повторном сообщении об оплате обновить тот же заказ, а не создать второй | `order_id` |
| Подписка | Получить действующую подписку и выдать доступ до даты окончания периода | `subscription_id` |
| Сервер | После выбора тарифа найти сервер по заказу; создать один раз | `order_id` или `server_id` |
| Группа доступа | Добавить сотрудника после назначения роли и убрать после отзыва роли | `user_id` + `group_id` |
| Товар или тариф | При синхронизации обновить существующий объект или создать отсутствующий | `external_product_id` |
| Счёт | По повторному сообщению о платеже отметить тем же платежом один и тот же счёт | `payment_id` |
| Чат или тикет | Найти диалог по внешнему ID клиента; создать только для нового обращения | `external_chat_id` |

Во всех случаях порядок один: найти по устойчивому ключу → убедиться, что записи нет или она изменилась → создать или обновить → сохранить связь между системами.

## Доступ — отдельная операция

Создать техническую сущность и выдать доступ — не одно и то же. Источником платного доступа должна быть подтверждённая, относящаяся к нужному сервису подписка или политика доступа. Не используйте строку в истории платежей как единственное доказательство доступа.

Подписка со статусом отмены может оставлять доступ активным до `current_period_end`: автопродление отключено, но уже оплаченный период ещё действует.

## Пример «Твой VPS»

1. Кабинет сопоставляет пользователя PasarGuard с неизменяемым ID контрагента Gigma; номер телефона использует только как дополнительный признак.
2. Создаёт бесплатную техническую учётную запись, только когда специальный endpoint поиска Gigma или PasarGuard однозначно подтвердил отсутствие записи.
3. Gigma возвращает действующую VPS-подписку.
4. Кабинет назначает тариф и срок тому же пользователю PasarGuard.
5. Повторная сверка с Gigma восстанавливает корректное состояние после задержек и временных ошибок.

Для этого проекта источник доступа — подтверждённая VPS-подписка из Gigma. Правило о конкретном webhook зависит от проекта: использовать событие можно только при проверяемой идентичности, защите от повторов и явно определённом источнике истины.
