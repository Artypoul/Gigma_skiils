# Создание тенанта в Gigma ERP

Тенант (проект) — изолированное пространство одного бизнеса. Всё в системе разделено по `project_id`.

## Эндпоинт

```
POST https://api.gigma.ru/api/projects        (публичный, токен НЕ нужен)
Content-Type: application/json
{ "name": "<Название>", "login": "<email-владельца>", "menu_id": 7 }
```

- `name` — обязательно.
- `login` — email будущего владельца. **Уникален на всю базу** (`users.login_unique`): один email = один пользователь во всей системе.
- `menu_id` — обязателен, должен существовать в `menus` (рабочее значение — `7`).
- Ответ `201` → `{"item":{"id":<PID>,"name":...}}`.

## Что создаётся (после PR #243/#244 — тенант ПУСТОЙ)

- проект + **владелец-пользователь** (роль `owner`, все права);
- **6 ролей** (Собственник/Админ/Менеджер/Руководитель отдела/Сотрудник/Руководитель направления) и **5 отделов** — «скелет», заводится всегда;
- **никакого демо-каталога** — клон проекта-образца №18 убран (PR #244);
- **никакого Timeweb-сервера** — `domain` остаётся пустым (PR #243). Витрина живёт на общем headless-API.

Каталог, склад и витрину заводят отдельно и осознанно (склад требует `city_id`, витрина — меню/категории, иначе фронт ломается).

## Перед созданием

- **Email занят?** Логин глобально уникален — если у `login` уже есть пользователь, создание упадёт. Сначала освободи: удали старый проект этого юзера (через БД, см. ниже) или смени ему email.

## Проверка результата (через БД)

Доступ к боевой БД — только с **VPN-адреса** (нода PasarGuard `31.77.160.204`); хост `161.104.46.126` (priscilla), БД `default_db`, юзер `gen_user`, пароль — в `.env` репо `itecho-erp-backend` (`DB_PASSWORD`). Драйвер: `pip install pymysql`.

```sql
SELECT id,name,domain FROM projects WHERE id=<PID>;
SELECT id,login,project_id FROM users WHERE login='<email>';
SELECT COUNT(*) FROM nomenclatures WHERE project_id=<PID>;   -- должно быть 0
SELECT COUNT(*) FROM roles WHERE project_id=<PID>;           -- 6
```

## Удаление тенанта (если нужно пересоздать)

Безопасно: **бэкап → транзакция → удалить `brands` (это неткаскадный блокер) → `DELETE FROM projects WHERE id=<PID>`** (каскад уберёт владельца через `users.project_id` CASCADE и остальное). Перед commit сверить, что след пуст и владелец исчез. Всегда сначала дамп удаляемых строк в `.sql`.

## Если `POST /projects` отдаёт 500

Смотри причину в БД-логе: `SELECT message,created_at FROM activity_logs WHERE type='exception' ORDER BY id DESC LIMIT 5`. Исторические грабли: триггеры клонирования требовали `log_bin_trust_function_creators=ON` (после переезда БД), а Timeweb-провижининг падал и ломал откат на FK `brands`. Оба шага удалены — но смотри лог при любом новом 500.
