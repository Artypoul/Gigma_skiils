# FAQ patterns

FAQ should remove buying friction and implementation doubts. It is not a place to repeat marketing claims.

## Question sources

- sales calls;
- support tickets;
- objections from ads/search;
- competitor comparisons;
- pricing concerns;
- onboarding risks;
- legal/security concerns;
- timing and scope questions.

## Common FAQ categories

| Category | Example question | Purpose |
|---|---|---|
| Fit | Is this for [segment/use case]? | qualify users |
| Scope | What is included? | prevent mismatch |
| Timing | How long does it take? | reduce uncertainty |
| Price | What affects cost? | reduce price anxiety |
| Switching | Can we migrate from [alternative]? | reduce change risk |
| Proof | Have you done this for [industry]? | build trust |
| Security/legal | How is data handled? | reduce risk |
| Support | What happens after launch? | clarify ownership |

## Answer pattern

```text
Direct answer first. Then explain the condition or nuance. End with the next step if useful.
```

## Rules

- Keep answers specific and honest.
- Do not invent legal or technical guarantees.
- Use FAQ to name limitations, not hide them.
- If an answer needs proof, mark `[нужен proof]`.
- Avoid questions nobody asks, such as «Почему мы лучшие?».
